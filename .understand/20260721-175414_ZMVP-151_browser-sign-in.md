# 🔎 Understanding ZMVP-151 — A visitor signs in from the browser and signs out

> **Status:** In Progress · **Source:** https://zurnetwork.atlassian.net/browse/ZMVP-151 · **Generated:** 2026-07-21T17:54:14-06:00 · **Snapshot:** `.understand/20260721-175414_ZMVP-151_browser-sign-in.md`

## 🧭 1. Context (cold-start)

Epic **ZMVP-149 "The Other Side"** makes the SvelteKit app the first real client of the API. ZMVP-150 landed the workbench: `frontend/web` (Svelte 5 runes, adapter-node, vitest client/server projects), the Caddy one-origin split (`/api/*` → axum prefix-stripped; `/signin-callback`, `/.well-known/*`, `/plugin/v1/*` carve-outs verbatim to axum; catch-all → vite:5174), and the SSR half of the same split in `handleFetch` (`src/lib/server/api-proxy.ts` — rewrites `/api`, forwards **only** `zurfur.sid`).

ZMVP-151 wears The Who's OAuth machinery through real pages. Epic rulings that bind here: **(4)** SvelteKit owns the sign-in form, axum keeps the OAuth callback · **(9b)** no `/welcome` — first sign-in lands on `/` signed-in, which *is* the provisioning proof · **(1B)** crude-but-real, zero design work.

Backend reality (mapped from `crates/api/src/routes/session.rs`):
- `GET /` → bare-HTML sign-in page (`form`/`sign_in_page`) — **retires**.
- `POST /signin` → `auth.start(handle)` → 303 to the PDS authorize URL; failure re-renders the HTML page.
- `GET /signin-callback` → `auth.complete` → provision User (`uow.users().provision(&did)`, idempotent, ZMVP-123) → `cycle_id()` + session insert → **`Redirect::to("/me")`** (hardcoded, session.rs:240).
- `GET /me` → **HTML** greeting via `resolve_profile` (`Profile { handle, display_name, avatar_url }` — the data path the frontend needs already exists; only a JSON handler is missing). Anonymous → 303 `/`, no 401.
- `POST /logout` → `session.flush()` (server-side row delete + cookie drop, idempotent) → 303 `/`.
- Session: `zurfur.sid`, HttpOnly, SameSite=Lax, Secure in PROD/STG, 7-day sliding expiry, Postgres store — **consumed, never modified**.
- CSRF: `require_first_party_origin` — single-origin allowlist = `Config::public_url` (= Caddy origin :8080), state-changing methods only; no-Origin (SSR Node fetch) passes.
- Errors: RFC 9457 `Problem` (`crates/api/src/problem.rs`) — `type` URN `urn:zurfur:error:*` + `code` + `title`/`detail`/`status`; clients branch on `code`. The session HTML surface predates it.

The `~/code/zurfur-web` spike (reference crib) proves the patterns but built none of the seams: no typed client, no problem parsing, no hooks/locals/guards, no avatar. Reusable ideas: server-proxy rationale, cookie-passthrough session check, name-driven logout cookie-clear. Its README flags exactly the two backend gaps this ticket must close: **no JSON `/me`** and **callback lands on backend `/me`**.

## 🗺️ 2. Domain

- **User as actor** (DD 26247170, actor super-table DD 34013187): first sign-in interns the DID into `actor_identity` + lands the `users` projection — idempotent provisioning, "registration is provisioning theater."
- **Auth surfaces & CSRF** (DD 24543244): cookie BFF surface behind the one-origin allowlist; `/plugin/v1` bearer surface untouched here. Nothing in this ticket may loosen the boundary.
- **API response shape** (DD 23592962): bare success bodies + problem+json errors — the convention the new frontend rendering seam consumes.
- **Handle resolution** (DD 26607618): `/.well-known/atproto-did` carve-out must keep reaching axum (already Caddy-guaranteed; don't disturb).
- Session/token handling ⇒ **security-review applies** to this ticket by the Definition-of-Done trigger list.

## 🎯 3. Goal & scope

**Goal:** a browser-only sign-in/sign-out loop through real SvelteKit pages — `/login` (handle → OAuth) → PDS → axum callback → signed-in `/` showing handle + avatar → `/logout` → signed-out `/` — while retiring the bare-HTML surface and establishing the two seams every later screen reuses (typed API client, problem-response rendering).

**In scope:**
- SvelteKit: `/login`, signed-in/out `/`, `/logout`, session guard (redirect signed-out session routes → `/login`), typed API client + problem parsing/rendering, header with handle+avatar.
- Backend (minimum to serve the above): JSON whoami, callback success redirect → `/`, sign-in/callback failure shapes that a JSON/redirect client can consume, retire the HTML pages (`sign_in_page`, `form`, `render_me`, `SIGN_OUT_FORM`).

**Out:** accounts/commissions UI (ZMVP-152/153) · any design/visual work (ruling 1B) · session mechanics changes (cookie, expiry, store, CSRF) · `/plugin/v1` · vanity routes · welcome flow (ruling 9b) · the old React `frontend/auth` app (stale, targets a removed token API — not touched, not referenced).

## 🧠 7. Logic & shape

```
Browser (origin :8080 = Caddy)
  GET /login ──catch-all──► SvelteKit page (handle input)
  POST /login (action) ──SSR fetch /api/signin──► axum POST /signin
        ◄─ 303 Location: PDS authorize URL ─┘   (redirect:'manual', relay via redirect(303))
  … PDS auth …
  GET /signin-callback ──carve-out──► axum: complete → provision → cycle+insert sid
        ◄─ 303 / ──────────────────────┘  (was /me)
  GET / ── layout load → /api/me (JSON) ──► signed-in page: handle + avatar
  POST /logout (action) ── /api/logout ──► flush → signed-out /
  GET /accounts (future, signed-out) ── guard ──► 303 /login
```

One code path both sides: pages always fetch `/api/...` — browser rides Caddy, SSR rides `handleFetch` (`api-proxy.ts` forwards only `zurfur.sid`).
